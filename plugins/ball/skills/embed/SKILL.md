# Ball Embed ‚Äî safe embedded execution of dynamically-delivered programs

Ball turns "download code and run it" from an arbitrary-code-execution problem into a bounded, auditable one. A Ball program is a Protocol Buffer message, not text: there is no `eval`, no FFI in the language, and **every** side effect flows through a named base function in a known module (`std`, `std_io`, `std_fs`, `std_memory`, `std_concurrency`, `std_collections`, `std_convert`, `std_time` ‚Äî CLAUDE.md invariant #4). That closed side-effect channel is what makes static audit *provably complete* and what lets you hand a program only the capabilities you choose.

This skill teaches the embedding pattern and its non-negotiable security invariants. It is grounded in the Ball repo; where a target's safety knobs are missing or immature, it says so instead of pretending parity. Read `references/embedding-per-target.md` and `references/audit-and-policy.md` for the copy-ready, verified code ‚Äî the invariants below are only safe when you use that code, not a paraphrase of it.

## The core pattern: receive ‚Üí audit ‚Üí reject-or-execute ‚Üí vetted modules

```
untrusted bytes
  ‚îÇ
  ‚îú‚îÄ 0. VERIFY AUTHENTICITY   (signature/hash over the EXACT raw bytes, BEFORE any decode)
  ‚îú‚îÄ 1. SIZE-GATE the raw received bytes   (byte length, BEFORE any decode)
  ‚îú‚îÄ 2. decode ONCE to a Program/Module    (transport: .ball.bin or .ball.json)
  ‚îú‚îÄ 3. AUDIT that same decoded object
  ‚îÇ        ‚Ä¢ capability policy (deny fs/process/memory/concurrency ‚Äî + io/time/random for a pure decision)
  ‚îÇ        ‚Ä¢ your OWN isBase-module allowlist   (audit can't see custom-module DECLARATIONS)
  ‚îÇ        ‚Ä¢ reject any non-empty module_imports; never pass a `resolver`
  ‚îÇ        ‚Ä¢ termination check (Dart + compiled TS CLI; heuristic, not a halting proof)
  ‚îú‚îÄ 4. REJECT on any violation ‚îÄ‚îÄ or ‚îÄ‚îÄ‚ñ∂ EXECUTE that SAME object, locked down
  ‚îÇ                                          ‚Ä¢ try/catch the CONSTRUCTOR *and* run()/callFunction()
  ‚îÇ                                          ‚Ä¢ sandbox: true + every resource limit set
  ‚îÇ                                          ‚Ä¢ Dart: DENY-BY-DEFAULT module allowlist (StdModuleHandler.subset + only vetted modules)
  ‚îÇ                                          ‚Ä¢ bound host-side dispatch frequency for long-lived UI loops
  ‚îî‚îÄ output   (run()/callFunction return value / injected stdout)
```

The Dart `ball audit` CLI verb (`_audit`, `dart/cli/lib/src/runner.dart:460`) is the reference implementation of the audit half; its own `run` path (`_run`, `runner.dart:312`) is the minimal embedding it guards. **The audit and the engine sandbox are two different layers** ‚Äî audit is a static pre-filter, the engine's `sandbox`/module-allowlist is run-time enforcement. You need both; neither substitutes for the other. As of #402 the audit categorizes a call by the base-function identity the engine *dispatches* (a bare-name fallback), not by the spoofable call-site `module` string ‚Äî so `{module: "harmless", function: "mutex_create"}` is now denied as `concurrency`, making audit-then-execute sound against call-level module spoofing. It stays a static pre-filter, not the whole boundary (see the residual under "Security invariants").

## Security invariants (non-negotiable checklist)

- [ ] **Audit the exact bytes you execute.** Decode once, audit that object, execute that object. Never audit one payload and run another (TOCTOU). If you must shell out to a file-based `ball audit` (the only option on TS), write the received bytes to a temp file and load *that same file* into the engine ‚Äî see the TS recipe in the reference.
- [ ] **Reject remote and embedded imports.** The engine takes an optional `ModuleResolver? resolver` (`engine.dart:201/222`) and a program's modules can carry `module_imports[]` with `http`/`git`/`registry`/`inline` sources (`ball.proto:148-162`) ‚Äî a built-in code-import vector that needs no custom module. `analyzeCapabilities` does **not** walk import metadata, so a passing audit says nothing about them. For untrusted input: **never pass a `resolver`, and reject any program whose modules have non-empty `module_imports`** (inline embeds un-audited code; http/git/registry fetch remote code). Runnable check is in the reference.
- [ ] **Deny-by-default module surface ‚Äî but only Dart enforces it at run time.** On Dart, register `StdModuleHandler.subset({‚Ä¶})` (an allowlist of only the pure/UI functions you need) plus your own vetted modules, and never register `std_fs`/`std_memory`/`std_concurrency` handlers. **On TS you cannot do this:** `new BallEngine(...)` always constructs a fresh, full `StdModuleHandler` (`ts/engine/src/index.ts:83,114`) with no injection point, so every implemented `std_*` function stays reachable at run time. The audit + your own allowlist walk is the *only* gate on TS ‚Äî treat it as such.
- [ ] **`sandbox: true` is not sufficient alone.** It blocks only `std_fs.*` + `std_io.exit`/`panic`/`env_get` (verified: `engine_std.dart` `_checkSandbox` call sites). It does **not** block `std_concurrency.*` (implemented ‚Äî `thread_spawn`/`mutex_*`/`atomic_*` at `engine_std.dart:1506-1536` ‚Äî and runs freely under sandbox). `std_memory.*` currently throws `Unknown std function` in the Dart engine (unimplemented on this branch ‚Äî a coverage gap, **not** a deliberate block; do not rely on it). Deny `memory`/`concurrency` at audit time *and* (on Dart) keep them out of your `subset`.
- [ ] **The audit cannot see custom-module *declarations* ‚Äî but call-site module spoofing is now categorized.** `analyzeCapabilities`/`ball audit` classifies base functions from `capability_table.dart`. Two things still get through and one no longer does:
  - **Still blind:** a program that **declares its own `isBase` module** (an attacker's `evil.exfiltrate`, or your own `ui`) is reported **pure / no risk**. You MUST add your own walk over `program.modules[]` that rejects any `isBase` function whose module name is outside a strict, explicit allowlist ‚Äî allowlist the *specific* module names you register, not "any `std*`-looking name."
  - **Still blind:** `module_imports` (next bullet).
  - **Now caught (#402):** a bare *call* to a real base function under a lying `call.module` string (`{module: "harmless", function: "mutex_create"}`) is resolved by its globally-unique bare name and categorized (`concurrency`) regardless of the spoofed module, on both the Dart audit and the compiled TS CLI. **Residual:** if the program also *declares* a same-named non-base **decoy** user function, audit treats the call as that user function and skips the bare-name resolution; whether that becomes a real bypass depends on engine dispatch order (engine-side fail-loud on cross-module bare-name base resolution is deferred hardening per #402). So the strongest posture is **audit + a deny-by-default module allowlist on the engine (Dart), not audit alone** ‚Äî never register a handler for `std_io`/`std_fs`/`process`/`memory`/`concurrency`/net for untrusted programs. The engine failing closed on an unregistered module handler is the real boundary ‚Äî but that backstop evaporates the moment you register an over-broad handler (`handles(_) => true`), so never write one.
- [ ] **Set every resource limit explicitly, and try/catch the constructor.** Defaults target trusted source (TS: `maxModules`/`maxExpressionDepth` = 1,000,000; `maxRecursionDepth` = 100,000). Set `timeoutMs`, `maxMemoryBytes`, `maxModules`, `maxExpressionDepth`, `maxRecursionDepth`, `maxProgramSizeBytes` yourself, low. `maxProgramSizeBytes`/`maxModules` are validated in the **constructor** (`_validateProgramLimits`, `engine.dart:229/247`) and throw synchronously ‚Äî wrap the `BallEngine(...)` construction, not just `run()`, or an oversized payload crashes your process instead of being rejected.
- [ ] **Size-gate raw bytes before decoding.** Gate the *received byte count* (e.g. the `Uint8List`/`Buffer` length, or `utf8.encode(json).length`), not a decoded string's `.length` ‚Äî a string's length counts UTF-16 code units, so a payload of multi-byte characters can be ~3√ó its real wire size and slip a length check while the expensive UTF-8 decode has already run. `maxProgramSizeBytes` fires only *after* decode, so it is a second layer, not the first.
- [ ] **All program content is untrusted ‚Äî not just metadata.** The engine never branches on `metadata` (CLAUDE.md invariant #2), so you never strip it for execution safety, but it carries attacker-controlled strings. The same is true of every value a program passes *into* your vetted modules (a `ui.text` string, a `db.query` argument). Never render metadata *or* content into UI/HTML/logs unescaped, and never branch privileged app logic on either. Crucially, do not implement a UI primitive with an HTML/Markdown/WebView widget: a `ui.text` backed by a web view turns attacker content back into interpreted markup and reopens the exact RCE surface this pattern removes. Use plain text/native widgets with escaped strings.
- [ ] **A vetted module must stay side-effect-free / rendering-only.** A `ui`-style module is a hard capability boundary: expose only pure render/return primitives (`text`, `button`, `column`) and never let it grow effectful members (`ui.open_url`, `ui.http_get`, `ui.write_file`). Scope creep in your own handler silently re-adds a capability the audit can never see (it is a custom module ‚Äî a permanent blind spot). Every function in it needs a `default: throw` and a specific `handles()`.
- [ ] **Bound host-side dispatch frequency for long-lived loops.** `timeoutMs`/`maxRecursionDepth`/`maxMemoryBytes` bound a *single* `run()`/`callFunction()`. Nothing in the engine bounds how often *your host* re-invokes `callFunction` ‚Äî a dynamic-UI event/re-render loop whose handler always requests another render can livelock outside every per-call limit. Add a host-level budget (max dispatches/second, or a max-renders-per-frame cap that drops rather than crashes).
- [ ] **A dedicated engine instance per untrusted run.** Never reuse an engine that has privileged first-party handlers (analytics, storage, navigation) to run server-pushed programs.
- [ ] **Authenticity is your job, not Ball's.** Ball does not sign or authenticate payloads. Sandboxing limits what a program *does*, not whether it is the one your server sent ‚Äî a compromised CDN/cache or a replayed, since-revoked version substitutes a still-audited, still-sandboxed, but attacker-chosen program. **Verify a signature/hash over the exact raw bytes as step 0, before decode/audit/execute** (worked Ed25519 example in `references/embedding-per-target.md`), and treat structurally-valid ‚âÝ safe *business logic*.

## The extensibility seam: expose ONLY a vetted native surface

Native capabilities (UI primitives, a `db.query`, a `nav.push`) enter an engine exclusively through a `BallModuleHandler` you write (`dart/engine/lib/engine_types.dart:403`). That is the same mechanism you must never hand carelessly to untrusted code ‚Äî and exactly how you give a dynamic-UI program a safe, minimal API:

```dart
class UiModuleHandler extends BallModuleHandler {
  @override bool handles(String module) => module == 'ui';   // NEVER `=> true`
  @override Object? call(String fn, Object? input, BallCallable engine) => switch (fn) {
    'text'   => renderText(input),
    'button' => renderButton(input),
    _        => throw BallRuntimeError('Unknown ui function: "$fn"'),  // fail closed
  };
}
```

Only plain data crosses the boundary ‚Äî and it is untrusted (escape every string; back the primitive with a native widget, never a WebView). Keep this module rendering-only: no `open_url`/`http_get` members. Event handlers are safest as `{module, function}` **name pairs** re-invoked later ‚Äî a tap re-enters the same already-audited program ‚Äî rather than live Ball closures held across the boundary. Bound how often your host re-dispatches those handlers (see the dispatch-budget invariant).

**Producing the content that calls `ui.text(...)`.** The `ui` module must be *declared* inside the program (an `isBase: true`, no-body stub per function) so your allowlist walk can see it, and the server-authored function bodies must emit `FunctionCall`s to it. The Dart/TS **source encoders route everything through universal `std`** ‚Äî they have no convention that emits a call into a custom `ui` module ‚Äî so author custom-module programs with the proto builders (`package:ball_base` types in Dart, `create(ProgramSchema, ‚Ä¶)`/`fromJson` in TS), not the source encoder. A verified proto-builder + proto3-JSON authoring snippet is in `references/embedding-per-target.md`.

### Passing data in and getting a decision out

Two verified I/O channels (Dart; the TS/Rust wrappers capture stdout as a `string[]` return):

- **`run()`** ‚Üí `Future<Object?>` returns the **entry function's** return value (a Ball value: `Map`/`List`/scalar). Program `std.print` output is delivered via the injected `stdout` callback, *not* `run()`'s return. The constructor's `args` is `List<String>?` ‚Äî CLI-argv-shaped, not a channel for structured business context.
- **`callFunction(String module, String function, Object? input)`** ‚Üí `Future<Object?>` (`engine.dart:244`) invokes a named function with a structured `input` and returns its result. This is the channel for a rules engine: pass your context (`{'cartTotal': 42.0, 'tier': 'gold'}`) as `input`, read the structured decision back from the return value. Re-invoking event handlers by name pair uses the same method.

## The three scenarios

- **Server sends dynamic instructions (rules engine).** Author the program from real source with a Ball **encoder** (`ball encode <source> -o prog.ball.bin`; `ball_encoder` for Dart source, `@ball-lang/encoder` for TS source) ‚Äî do not hand-write proto3-JSON. Gate it at CI time with `ball audit --deny fs,process,memory,concurrency,io,time,random --exit-code` (see references ‚Äî deny alone exits 0 on the Dart CLI). That is the **canonical deny-set for a pure, deterministic decision**: `fs,process,memory,concurrency` is the security-critical core (host compromise / runaway compute); adding `io,time,random` forces the rule to be reproducible ‚Äî permit `io` back only if the program legitimately prints, `time`/`random` only if the decision genuinely needs a clock/RNG. Sign the exact `.ball.bin` bytes, ship behind your authenticated API; the client verifies the signature, re-audits, and executes locked-down, reading the structured decision back via `callFunction` (one combined worked example in the reference). Ball is not needed at request-serving runtime ‚Äî only to produce and vet the blob.
- **Client audits at runtime, then executes.** On Dart, call `analyzeCapabilities`/`checkPolicy`/`analyzeTermination` from `ball_base` in-process before constructing the engine. On TS there is no exported in-process capability library (`@ball-lang/cli`'s package `exports` genuinely block importing its analyzer) ‚Äî do your own module-allowlist walk in-process, and/or shell out to the `ball audit` CLI subprocess. **Invoke it as `node <resolved cli/dist/index.js> audit ‚Ä¶`, not the extensionless `node_modules/.bin/ball` shim** ‚Äî that shim throws `ENOENT` under `child_process` without `shell: true` on Windows (see reference). Then execute on the locked-down engine.
- **Dynamic native UI.** Author the screen as a `Program` (or library `Module`) that declares a vetted `ui` module (`isBase` stubs) and function bodies that call `ui.text(...)`/`ui.button(...)` ‚Äî built with the proto builders, since the source encoders only emit `std` (authoring snippet in the reference). **A bare `Module` is not directly runnable ‚Äî `BallEngine` takes a `Program`.** Audit a `Module` with `analyzeModuleCapabilities(module, imports: ‚Ä¶)` (not `analyzeCapabilities`, which is typed to `Program`), then wrap it into a `Program` (set `modules`, `entryModule`, `entryFunction`) to execute ‚Äî see the reference for both snippets. Register only your `UiModuleHandler` (plus, on Dart, a `std` subset); a `default: throw` inside it is a hard allowlist. Keep that handler rendering-only, treat every value it receives as untrusted (native widgets, no WebView), and bound host-side re-render dispatch. This gives truly native dynamic UI with no arbitrary code execution ‚Äî **Dart only** today (TS has no run-time custom-module registration).

## Per-target honest status (read the reference before you pick one)

| Target | Embeddable for untrusted input? |
|---|---|
| **Dart** (`ball_engine` + `ball_base`, pub.dev) | **Yes ‚Äî reference target.** Full sandbox + all limits + `moduleHandlers` (runtime std allowlist) + in-process audit library. Caveat: `ball_engine` pulls `dart:io` (via `ball_resolver`) and **does not compile for Flutter/Dart web** ‚Äî engine embedding is mobile/desktop/server only; `ball_base`'s audit library *does* support web, so a web client can audit but must run the program elsewhere. |
| **TypeScript** (`@ball-lang/engine`, npm) | **Partial ‚Äî audit-gated only.** Sandbox + resource limits work, but you **cannot restrict the std surface or add custom modules at run time** (constructor hard-codes a full `StdModuleHandler`). Safety rests entirely on your pre-execution audit/allowlist walk + rejecting `module_imports`. The `ball audit` CLI is now **self-hosted** (compiled from the same `cli_core.dart`, #362/#398) ‚Äî so the #402 call-spoofing fix and termination analysis apply to it too ‚Äî but there is still **no in-process audit library** (spawn the CLI via `node <cli/dist/index.js>` or do your own walk). No custom-native-`ui`-module story yet. |
| **Rust** (`ball-engine`) | **Trusted only, and not on crates.io.** Runs at Dart parity, but the public API exposes no sandbox/limit/custom-module knobs. **Do not `cargo add ball-engine` ‚Äî that name on crates.io is an unrelated 2D physics engine.** Vendor via a git dependency on `Ball-Lang/ball` (see reference). |
| **C++** (`engine_rt`, vendored) | **Trusted only.** No package; vendor + regenerate + `#include`. No demonstrated sandbox path. |
| **C#** | **No.** Engine does not execute to golden output yet (`SelfHostPendingException`). Encoder + value model only. |

**Read `references/embedding-per-target.md`** for verified, copy-ready snippets per target (constructor surfaces, exact defaults, the full receive‚Üíaudit‚Üíreject‚Üíexecute flow, and every gap above with its evidence). **Read `references/audit-and-policy.md`** for capability categories, the `--deny`/`--exit-code` semantics (which differ between Dart and TS), what termination analysis does and does not prove, and why `network` is a vacuous category today.

## Dangerous assumptions (grounded in real evaluation failures)

| Assumption | Reality |
|---|---|
| "`sandbox: true` blocks everything dangerous." | It blocks fs + `exit`/`panic`/`env_get` only. `std_concurrency.*` is implemented and runs freely under it ‚Äî deny it at audit time and (Dart) keep it out of your `subset`. `std_memory.*` happens to throw `Unknown std function` on the current Dart engine, but that is an unimplemented coverage gap, not a guarantee ‚Äî audit-deny `memory` regardless. |
| "`StdModuleHandler.subset({‚Ä¶})` locks down the TS engine too." | No. `new BallEngine(...)` in TS always builds a full, unrestricted handler and offers no way to inject a subset ‚Äî the exported `subset()` you construct is never wired in. Runtime std restriction is a **Dart-only** control. |
| "`ball audit` passing means the program is safe." | It is now sound against call-level module spoofing (#402: a base call categorized by resolved bare-name identity, not the spoofable `call.module`), but still blind to custom `isBase` module *declarations* (reports them pure) and to `module_imports`. Add your own module-name allowlist walk and reject non-empty imports; keep a deny-by-default engine module allowlist (Dart) so the residual decoy-user-function path and any future gap can't reach a dangerous handler ‚Äî and never register a `handles(_) => true` handler that defeats fail-closed. |
| "The engine defaults are safe for untrusted input." | Defaults target trusted source and are very permissive (esp. TS: 1,000,000 module/depth caps). Set every limit yourself, and wrap the **constructor** in try/catch ‚Äî size/module limits throw there, not from `run()`. |
| "`cargo add ball-engine` installs the Ball engine." | It installs someone else's 2D physics crate (crates.io `ball-engine` v0.1.1, `github.com/parth2152012/ball-engine`). No Ball Rust crate is published; use a git dependency and verify provenance before adding it. |
| "`ball audit --deny fs,network` fails the build on a violation." | On the Dart CLI, `--deny` alone prints and exits **0**; you must add `--exit-code`. The TS CLI always exits 1. They are not at parity. |
| "`--deny network` protects me from network access." | No base function maps to `network` today ‚Äî the guarantee is vacuous. Network can only ever enter via a custom module *you* write (or an unrejected remote `module_import`), so gate those seams. |
| "Denying `io` is harmless / narrow." | `std.print` maps to the `io` capability ‚Äî deny `io` and a legitimate `print`-only program is rejected. Choose deny sets deliberately; a too-broad deny is a silent availability bug, a too-narrow one a security hole. |
| "Structurally valid / audited ‚áí correct." | Audit and sandbox stop host compromise and runaway compute, not a wrong *decision* ("approve every refund"). Business logic still needs its own review and a server-side kill switch. |
