# `ball-engine` — Rust Ball engine (self-hosted)

Runs Ball programs by the **self-host** route (SKILL.md Phase 4, Option B), the
same approach as the TS and C++ targets: the reference engine is itself a Ball
program (`dart/self_host/engine.ball.json`), compiled through `ball-compiler`
into `src/compiled_engine.rs`, and driven by a thin native wrapper.

## Layout

- `src/lib.rs` — public API (`BallEngine::from_json` / `from_binary` / `run`)
  and `EngineError`.
- `src/loader.rs` — load a target `Program` from proto3-JSON `.ball.json` or
  binary `.ball.bin` (via `prost-reflect`), plus the canonical proto3-JSON
  `BallValue` view the compiled engine reads.
- `src/scope.rs` — lexical `Scope` chain (`Rc<RefCell>` frames, parent links)
  and `FlowSignal` (return/break/continue/throw).
- `src/ball_proto.rs` — native impls of the `ball_proto` access-pattern
  functions (oneof discriminators `whichExpr`/…, presence checks `hasBody`/…,
  safe field get/set) + virtual properties (`.length`/`.isEmpty`). Semantics
  match `dart/shared/lib/ball_proto.dart` exactly.
- `src/compiled_engine.rs` — **GENERATED, NEVER EDIT** (see below).
- `tool/` — the `ball-engine-regen` crate (regeneration entry point).

## Generated files — NEVER edit

- **`src/compiled_engine.rs`** — the self-hosted engine, compiled from
  `dart/self_host/engine.ball.json`. Regenerate, never hand-patch. It is
  git-ignored (like the C++ self-host's `engine_rt.cpp`) because it is a
  ~19k-line build artifact. To change it, fix `rust/compiler/` (or the
  `dart/self_host/` source) and rerun the regenerator.

## Regenerate the self-hosted engine

```bash
cargo run -p ball-engine-regen
```

Loads `dart/self_host/engine.ball.json`, strips the `@type` Any envelope,
compiles via `ball_compiler::Compiler::compile_library()` (library mode — the
engine has no runnable top-level entry point; its public surface is the
`BallEngine`/`StdModuleHandler` classes), and writes `src/compiled_engine.rs`.
`engine.ball.json` is itself git-ignored and regenerated by
`dart/compiler/tool/gen_engine_json.dart` (CI regenerates it before this step).

## Self-host status (issue #39) — COMPILES; run blocked on `this` semantics

The regeneration pipeline is complete and the wrapper foundation
(loader + scope + `ball_proto` access patterns) builds and is unit-tested.
**`compiled_engine.rs` now builds cleanly through `ball-compiler`** — `cargo
build -p ball-engine --features self_host` succeeds with **0** errors (down from
~326 → ~348 → 186 → 25 → **0**). The compiled-engine *driver* is still gated
behind the off-by-default `self_host` feature and `BallEngine::run` returns
`EngineError::SelfHostPending` in the default build, because the compiled engine
**does not yet run correctly** (see "Run blocker" below). `cargo build -p
ball-engine` / `cargo test -p ball-engine` stay green on the foundation.

### Run blocker — value-vs-reference `this` semantics (the next epic)

The self-hosted engine is a tree-walking interpreter that relies on a **mutable
`this`**: `_buildLookupTables` fills `this._functions`, `run` then reads it, the
scope chain mutates `this._globalScope`, etc. This crate's compiled model
represents an instance as a **by-value** `BallValue::Message`, so a method's
`let self_ = ball_field_get(input, "self")` is a *clone* — every mutation of
`this.<field>` (or a field-alias local) lands on that clone and is **lost** when
the method returns. So even though every function compiles and type-checks,
`run()` would read an empty `_functions` table and throw "entry point not
found". Two coupled gaps remain, in dependency order:

1. **Implicit-`this` method dispatch.** A `this.method(...)` call is encoded
   with only its arguments; the compiler emits `method(<args>)` with no `self`
   in the argument message, so the dispatcher's `ball_field_get(input, "self")`
   is `Null` and the method panics on the first call. (A prototype injecting
   `self_` here was reverted: it fires incorrectly in methods that *shadow*
   `self_` with an unrelated local — `let mut self_ = unwrappedSelf` — and
   introduces `&mut self_` / `self_.clone()` borrow conflicts, all for **no
   payoff** while gap 2 blocks execution.)
2. **Reference semantics for `this`** — the real blocker. Making mutations
   persist needs a shared, interior-mutable instance representation (an
   `Arc<Mutex<…>>`-backed `Map`/`Message` variant, since `BallValue` must stay
   `Send` for `ball_throw`'s `panic_any`). That reshapes `BallValue`'s entire
   value model — every clone, mutation site, `Display`, and `PartialEq`, plus
   the compiler's `&mut self_` borrow emission — and changes aliasing semantics
   for *all* programs (risking the passing conformance corpus). It is its own
   architectural epic, not a self-host follow-up.

Until those land, `run` stays `SelfHostPending`. The acceptance programs
(hello_world/fibonacci/factorial/string_ops + closures + map-order) therefore
do **not** yet run through `ball-engine`.

### Resolved compiler gaps (25 → 0)

The final batch (this pass) cleared gap #5 (class hierarchy / super-constructor
/ constructor-body) plus the last Dart-SDK helpers. The full history (each a
former blocker):

1. **Protobuf oneof-discriminator enum constants** — RESOLVED (71 errors, was
   ~70). The engine compares `whichExpr()`/`whichValue()`/`whichStmt()`/
   `whichKind()`/`whichSource()` to `Expression_Expr.call`, `Statement_Stmt.let`,
   `Literal_Value.stringValue`, `structpb_Value_Kind.*`, `ModuleImport_Source.*`.
   These are synthesized oneof discriminators with no `EnumDescriptorProto`, so
   the compiler emitted an unresolved `Expression_Expr` name. Fixed by
   `type_emit::oneof_discriminator_enum_defs` (wired into `Compiler::compile`),
   which emits each as a crate-root `pub static <Enum>: LazyLock<BallValue>`
   namespace whose members are the oneof arm's **string** case name (matching
   the `ball_proto` discriminator return values) — so `Expression_Expr.call`
   resolves to `BallValue::String("call")`. Mirrors the TS compiler's
   `preamble.ts` constants. (`io_FileMode` — a `dart:io` enum, not a Ball oneof
   discriminator — is now emitted as a runtime static alongside gap #2.)
2. **Dart-SDK method calls** — RESOLVED for the whole String/List/Map/Set/`num`/
   `int`/`double`/`String`/`DateTime`/`RegExp`/`File` surface (~140 errors, was
   ~120 estimated). Fixed in `ball_shared::runtime` (the `dartsdk` submodule,
   re-exported via `pub use dartsdk::*`), which defines each Dart-SDK method as a
   free `fn <name>(input: BallValue) -> BallValue` matching the compiler's
   emitted call shape — a method's receiver rides in the packed `input` message
   under `self`, positional args as `arg0`/`arg1`/…, named args by name (the
   invariant-#1 convention, see `ball_arg_get`). Implemented with exact Dart
   semantics: `tryParse`/`parse`/`remainder` (numeric, `self` = an `int`/`double`/
   `num` **type marker** value — also new runtime statics), `firstMatch`/
   `allMatches`/`group` (regex, via the new `regex` dep — a `RegExpMatch` is a
   `{groups: [...]}` message), `addAll`/`remove`/`clear`/`setAll`/`cast`/`toSet`/
   `unmodifiable`/`filled`/`elementAt` (collections), `union_`/`intersection`/
   `difference` (sets, `List`-backed per issue #35), `fromCharCode`, `identical`
   (structural — a by-value model has no object identity), `DateTime.now`
   (`{millisecondsSinceEpoch,…}`), the `dart_math::*` trig/log/exp/`atan2`
   functions, and the `dart_io::File`/`Directory` + `existsSync`/
   `readAsStringSync`/`writeAsStringSync`/`writeAsBytesSync`/`createSync`/
   `deleteSync` filesystem ops (+ the `io_FileMode` static). The module-qualified
   `dart_math::`/`dart_io::` calls resolve via a small compiler change: an
   **empty namespace module** (no Ball functions/types — a foreign-SDK marker)
   now emits `pub use ball_shared::runtime::*;` instead of a private
   `use super::*;`. Also fixed the missing `ball_null_coalesce` (`??`/`??=`)
   runtime helper the compiler had always emitted without a definition.
   **Still open** (Dart-SDK methods that need dynamic function dispatch — really
   gap #6, not a pure runtime helper): `Function.apply` (3) and `List.indexWhere`
   (1) invoke a `BallValue::Function` value, for which `ball-shared` has no
   dispatcher. The remaining `cannot find` errors are all out of this gap:
   `hasValue`/`hasObject`/`writeToBuffer` (proto-message methods, gap #3),
   `convert` (`JsonEncoder`/`JsonDecoder`, `std_convert`, gap #3), and
   `entries`/`self_`/`_initialized`/`stackTrace`/… (gap #4/#5).
3. **`ball_proto` / `std_convert` base dispatch** — the compiler emits
   `ball_unsupported_base_call("ball_proto", "whichExpr")`; it must route these
   16 `ball_proto` + 4 `std_convert` functions to runtime helpers (the wrapper
   already implements the `ball_proto` semantics in `src/ball_proto.rs`).
4. **Named/optional-parameter binding** — RESOLVED (22 errors). A multi-argument
   call packs its arguments into one `input` message (invariant #1) as `arg0`,
   `arg1`, … (positional) / `<name>` (named — `dart/encoder/lib/encoder.dart`'s
   `_encodeArgList` + `_setCallInput`); before this fix only a lone positional
   parameter was aliased (`let n = input.clone();`) and every other parameter was
   bound by its literal name, so positional arguments arriving as `argN` never
   resolved (`ballObjectSetField(target, fieldName, val)`, `_mathAtan2(a, b)`,
   `setField(name, value)` all `error[E0425]: cannot find value`). Fixed by
   `type_emit::params_binding_prologue` (shared by `method_prologue` and
   `param_alias_prologue`): a positional parameter binds via the new
   `ball_shared::runtime::ball_arg_get(input, name, "arg{i}")` runtime helper
   (prefers the parameter's name, falls back to its `arg{i}` slot — matching the
   TS self-host `_evalLambda` binding), a named parameter binds by name, and a
   receiver-less function's lone positional argument still binds directly. The
   remaining "getter" cases the old note lumped here are **not** parameter
   binding: `entries` is a field *inherited* from `BallObject`'s superclass
   `BallMap` (gap #5, super-inheritance), and `self_`/`enableProfiling`/
   `_callCounts`/`_initialized` are references in the `BallEngine.new` **constructor
   body** (also gap #5). `stackTrace` is a `try`/`catch (e, st)` secondary
   binding (a distinct try-handling gap).
5. **Class hierarchy / super-constructor / constructor-body** — RESOLVED
   (see the detailed entry after gap #6 below).
6. **Dynamic-value vs. borrow-checker** (E0618 / E0596 / E0499 / E0502 / E0308 /
   E0277 / E0382 / E0282) — RESOLVED (was ~150 errors; self-host total 186 → 25).
   The engine calls first-class function *values* and mutates `BallValue`s
   through the dynamic model in ways borrow-checked Rust rejected. Fixed by:
   - **First-class function values.** `BallFunction` is now a native-closure
     wrapper (`Arc<dyn Fn(BallValue) -> BallValue + Send + Sync>`, keeping
     `BallValue` `Clone`/`Send`/`Sync` — `Send` is required by `ball_throw`'s
     `panic_any`), invoked by the new `ball_shared::runtime::ball_call_function`.
     A `lambda` compiles to `BallValue::Function(BallFunction::new(…, move |input|
     …))`, and a top-level function referenced as a *value* is wrapped the same
     way (`compile_reference`). The 7 callback collection helpers
     (`list_map`/`filter`/… ) take a `BallValue` callback. Cleared E0618 (call a
     `BallValue`), the E0308 "found fn item"/"found closure", and E0277.
   - **Lexical scope tracking** (`Compiler::local_scopes`). Parameters, `let`s,
     `for`/`for_in`/`catch` vars, and a method's `self_`/field aliases are
     recorded as locals. This distinguishes a call *through a value* (a local
     `BallValue::Function` → `ball_call_function`) from a real function item — a
     user fn or a `ball_shared::runtime` Dart-SDK helper (`unmodifiable`/`now`/
     `cast`) → direct call — which a name-only test can't (a local can shadow a
     top-level namesake). It also guards `compile_reference`'s value-wrapping.
   - **Closure capture pre-cloning** (`compile_lambda` + `collect_referenced_names`).
     A `move` closure pre-clones each captured enclosing local so the original
     stays usable afterward (was E0382 "use of moved value").
   - **Mutation-borrow sequencing** (`emit_mutation`). The RHS value is bound to
     an owned temporary *before* the `&mut` slot borrow, and the slot is typed
     `&mut BallValue` (cleared E0499/E0502 and the `panic!`-slot E0282); a method
     field alias and a `let` mutated only inside a `switch`/`try` branch are now
     declared `let mut` (`expr_mutates_var` recurses into list-literal elements —
     cleared E0596).

   The remaining 25 errors were: E0425 (21) + E0790 (1) = gap #5
   (constructor-body / super-constructor inheritance / trait dispatch), and
   E0423 (2) = the Dart-SDK `StringBuffer.write` helper — **all resolved** (see
   gap #5 below).

5. **Class hierarchy / super-constructor / constructor-body** — RESOLVED
   (25 → 0). Four coordinated fixes in `rust/compiler` + `ball_shared::runtime`:
   - **Inherited-field binding** (`type_emit::all_instance_field_names` +
     `field_alias_prologue`, `Compiler::type_defs_by_short_name`). A
     method/constructor body's bare reference to an *inherited* instance field
     now binds like an own field: the compiler walks the `metadata.superclass`
     chain (resolved by short name) and, for a **native** base with no
     `TypeDefinition`, a small `native_superclass_fields` table
     (`BallObject extends BallMap` inherits the ordered-map backing `entries`).
     Cleared the 5 `entries` E0425s.
   - **Body-carrying constructors** (`compile_constructor_with_body`).
     `BallEngine`/`BallObject` constructors carry a real initializer-list + body;
     the synthesized fn now binds each parameter, builds `self_` (every instance
     field defaulted `Null`, overridden by `this.`-formals and by
     parameter-referencing field initializers — a richer initializer's cosmetic
     source-text value is best-effort `Null`), aliases every instance field,
     runs the body, and writes body-mutated fields back. Cleared the `self_` /
     `_callCounts` / `_initialized` / `enableProfiling` E0425s.
   - **Abstract classes emit a struct + inherent `impl`, not a `trait`**
     (`compile_type_def`). Dispatch is runtime `type_name` matching, so the
     `trait` was dead documentation — and a `trait`'s default method can't be
     called as `Trait::method(input)` from the dispatcher (E0790, the engine's
     inherited-but-not-overridden `BallModuleHandler::init`). As an inherent
     `impl` fn it is a real callable.
   - **`try/catch (e, st)` stack-trace variable** (`compile_try`) — the second
     catch variable is now bound (to the caught error's string form; the
     by-value model captures no real trace). Cleared the `stackTrace` E0425.
   - **Dart-SDK helpers** in `ball_shared::runtime::dartsdk`: `StringBuffer`
     (`write` + `.toString()` via a special-cased `ball_to_string`),
     `Function.apply`, `Iterable.indexWhere`, `dart:convert`
     `JsonEncoder`/`JsonDecoder.convert`, proto presence (`hasValue`/`hasObject`)
     and `writeToBuffer`. Cleared the `write` E0423s and the remaining
     proto-method / dynamic-dispatch E0425s.

A localized compiler fix landed with issue #39 earlier: the `__no_init__`
sentinel (uninitialized `late`/nullable declaration) now lowers to
`BallValue::Null` instead of an unresolved name — resolving 32 of the engine's
errors and a real bug for any program with an uninitialized nullable var.
