# Protobuf Editions Support for Ball's Publishable Protobuf Engine — Implementation Plan

> Status: PLAN — for review before any code is written.
> Sources cited inline; Ball file paths are real (verified against the repo).

---

## 1. Executive Summary

Ball already ships a **pure-Dart, descriptor-driven protobuf implementation** under `dart/shared/lib/protobuf/` (wire codecs, binary marshal/unmarshal, proto3-JSON codec, well-known types, gRPC framing) plus a **stub editions module** (`dart/shared/lib/protobuf/editions.dart`). Critically, `dart/encoder/bin/gen_ball_protobuf.dart` encodes this entire stack into a single Ball program (`dart/shared/ball_protobuf.json` / `.bin`) which Ball's compilers re-emit into every target language. This means a **single Ball-source protobuf runtime can be compiled to Dart, C++, TypeScript, Go, Python, Java, and C#** — most of whose native protobuf libraries either lag on Editions or never adopt it (e.g. Dart's `package:protobuf` exposes only generated bindings; C# explicitly never validates UTF-8 and treats all enums as open). We will make this engine **Editions-aware**: implement the full FeatureSet model and the canonical 5-step feature-resolution algorithm in Ball-portable Dart source, thread the resolved features through binary/JSON marshal and unmarshal, add proto2/proto3↔editions legacy inference, and prove portability by running the **same Ball editions engine on ≥2 targets** (Dart + TypeScript first, then C++) against golden FeatureSets generated by `protoc`. The payoff is the first concrete demonstration of Ball's portability thesis: write the hard semantics once in Ball, get a correct Editions-capable protobuf runtime everywhere.

## 2. Scope Decision — What "Ball's protobuf engine" Is

**Decision: extend the existing pure-Dart protobuf stack in `dart/shared/lib/protobuf/`, which is the delivery vehicle for the Ball-source `ball_protobuf` program.** This is grounded in current-state findings, not a greenfield invention:

- The stack is explicitly designed for this: `dart/shared/lib/protobuf/protobuf.dart` header says *"complete protobuf support written in pure Dart. When encoded to Ball IR, provides protobuf for any target language."*
- `dart/encoder/bin/gen_ball_protobuf.dart` already encodes all source files (including `editions.dart`) into `ball_protobuf.json` + `ball_protobuf.bin` via `DartEncoder`. That Ball program is the **publishable protobuf engine**.
- A stub resolver already exists (`editions.dart`) and is already wired into both the library barrel (`protobuf.dart`) and the encoder source list (`gen_ball_protobuf.dart`).

So the realistic delivery vehicle is **three layers, in order**:

1. **Author in Ball-portable Dart** under `dart/shared/lib/protobuf/` (the source of truth that gets encoded to Ball). Constraint: code must be encodable by `DartEncoder` and runnable by the engine — no constructs the encoder can't handle (top-level functions, `Map<String,Object?>`/`List<int>` data, no heavy class hierarchies).
2. **Encode to the Ball program** `dart/shared/ball_protobuf.json` / `.bin` via the existing generator.
3. **Compile/run on each target** through Ball's existing compilers and self-hosted engines.

**Explicitly out of scope (separate decision, see §7):** migrating `proto/ball/v1/ball.proto` itself from `syntax = "proto3"` to `edition = "2023"`. That is a *consumer-side* change with breaking-check and codegen ramifications across all 7 bindings; the engine work here does not require it and should not be coupled to it.

What we are NOT building: a new package, a new std module, or a protoc plugin. The engine is a library of Ball functions living inside the already-published `ball_protobuf` program.

## 3. Editions Feature Model — The Spec We Must Implement

Restated precisely from official sources:
- https://protobuf.dev/editions/overview/
- https://protobuf.dev/editions/features/
- https://protobuf.dev/editions/implementation/
- https://protobuf.dev/programming-guides/editions/
- `descriptor.proto` (main): https://raw.githubusercontent.com/protocolbuffers/protobuf/main/src/google/protobuf/descriptor.proto
- `feature_resolver.cc/.h` (main)

### 3.1 Edition enum (numeric, time-ordered)
`EDITION_UNKNOWN=0, EDITION_LEGACY=900, EDITION_PROTO2=998, EDITION_PROTO3=999, EDITION_2023=1000, EDITION_2024=1001, EDITION_2026=1002, EDITION_UNSTABLE=9999, EDITION_MAX=0x7FFFFFFF`. There is **no EDITION_2025**. `PROTO2`/`PROTO3` are internal sentinels (cannot appear in a file's `edition =` declaration); they exist so legacy files can be resolved through the same algorithm. In proto3-JSON the field serializes as the string name (e.g. `"EDITION_2023"`).

### 3.2 Core FeatureSet fields (the runtime-relevant ones we MUST implement)
All `RETENTION_RUNTIME`, all `edition_introduced = EDITION_2023`:

| # | feature | values | LEGACY | PROTO2 | PROTO3 | 2023 | 2024 |
|---|---------|--------|--------|--------|--------|------|------|
| 1 | `field_presence` | EXPLICIT / IMPLICIT / LEGACY_REQUIRED | EXPLICIT | EXPLICIT | IMPLICIT | EXPLICIT | EXPLICIT |
| 2 | `enum_type` | OPEN / CLOSED | CLOSED | CLOSED | OPEN | OPEN | OPEN |
| 3 | `repeated_field_encoding` | PACKED / EXPANDED | EXPANDED | EXPANDED | PACKED | PACKED | PACKED |
| 4 | `utf8_validation` | VERIFY / NONE (1 reserved) | NONE | NONE | VERIFY | VERIFY | VERIFY |
| 5 | `message_encoding` | LENGTH_PREFIXED / DELIMITED | LENGTH_PREFIXED | LENGTH_PREFIXED | LENGTH_PREFIXED | LENGTH_PREFIXED | LENGTH_PREFIXED |
| 6 | `json_format` | ALLOW / LEGACY_BEST_EFFORT | LEGACY_BEST_EFFORT | LEGACY_BEST_EFFORT | ALLOW | ALLOW | ALLOW |

`features` fields live at: FileOptions #50, MessageOptions #12, FieldOptions #21, EnumOptions #7 (plus oneof/service/method). `edition` is FileOptions #14.

### 3.3 Source-retention features (compiler-only; engine MUST ignore at runtime)
`enforce_naming_style` (#7, introduced 2024), `default_symbol_visibility` (#8, introduced 2024), `enforce_proto_limits` (#9, introduced 2026). All `RETENTION_SOURCE` → stripped from binary descriptors → **the engine carries none of these.** We will recognize-and-discard them, never act on them.

### 3.4 Language-specific extensions (engine MUST ignore gracefully)
FeatureSet extensions 1000–9994: `pb.cpp=1000, pb.java=1001, pb.go=1002, pb.python=1003, pb.csharp=1004`; third-party ≥10000. Ball's engine processes only base fields 1–9; **unknown FeatureSet extensions are skipped, never fatal.**

### 3.5 Field-level semantics the engine MUST honor
- **field_presence**: IMPLICIT → no `has_*`, default (0/false/""/empty) NOT serialized (proto3 singular scalar). EXPLICIT → presence tracked, set values always serialized. LEGACY_REQUIRED → must be present on parse/serialize. Only applies to singular non-repeated/non-map fields.
- **enum_type**: OPEN → out-of-range int kept in the field. CLOSED → out-of-range int routed to unknown-field set.
- **repeated_field_encoding**: PACKED → one LEN record. EXPANDED → one record per element. *Readers must accept both regardless of the feature; writers emit per the feature.*
- **utf8_validation**: VERIFY → reject non-UTF-8 string bytes. NONE → pass-through.
- **message_encoding**: LENGTH_PREFIXED → wire type 2. DELIMITED → group wire types 3/4.

### 3.6 The 5-step resolution algorithm (canonical, from feature_resolver.cc)
The data structure is `FeatureSetDefaults { repeated FeatureSetEditionDefault defaults; Edition minimum_edition; Edition maximum_edition; }` where each `FeatureSetEditionDefault { Edition edition; FeatureSet overridable_features; FeatureSet fixed_features; }`.

**File descriptor:**
1. Validate `minimum_edition ≤ file.edition ≤ maximum_edition` (EDITION_UNSTABLE allowed above max).
2. Binary-search `defaults` for the **highest** entry with `edition ≤ file.edition` (`upper_bound` then step back).
3. Base = `fixed_features` merged-under `overridable_features` (proto `MergeFrom` semantics: later set field wins; unset leaves prior).
4. Merge the file's explicit `options.features` on top.

**Non-file descriptor (message/field/enum/oneof/extension/service/method):**
1. Start from the **parent's already-fully-resolved** FeatureSet.
2. Merge this descriptor's explicit `options.features`.
3. Validate no `UNKNOWN` enum values remain.

**Parent rules (must be exact):** field → its oneof if in one, else its message; **extension field → its lexical enclosing scope, NOT the extendee**; nested message/enum → enclosing message else file; enum value → enum; method → service; file → edition defaults.

**fixed vs overridable:** a feature is *fixed* (user cannot override) when `edition < edition_introduced` OR `edition ≥ edition_removed`; otherwise *overridable*. Setting a fixed feature is a **hard error**.

### 3.7 Legacy inference (proto2/proto3 → features), before resolution
- `LABEL_REQUIRED` → `field_presence=LEGACY_REQUIRED`
- proto3 singular scalar (no label) → `field_presence=IMPLICIT`; proto3 `optional` → `EXPLICIT`; proto2 `optional` → `EXPLICIT`
- `TYPE_GROUP` → `message_encoding=DELIMITED`
- `[packed=true]` → `PACKED`; proto3 `[packed=false]` → `EXPANDED`
- proto2 file → enum_type=CLOSED, repeated=EXPANDED, utf8=NONE, json=LEGACY_BEST_EFFORT; proto3 file → the EDITION_2023 defaults table above (so a no-override `edition=2023` file == proto3).

## 4. Architecture

### 4.1 Components (all in `dart/shared/lib/protobuf/`)
1. **Edition model** — Edition enum int values + string⇄int mapping (`"EDITION_2023"` ↔ 1000), parsing the file `edition` declaration string (`"2023"`, `"2024"`) and the `syntax` string (`"proto2"`/`"proto3"`) into an Edition int. New file: `edition.dart` (or a section of the rewritten `editions.dart`).
2. **FeatureSet representation** — a feature set is a `Map<String,String>` (matches the existing stub's representation, and is encoder-friendly). Core keys = `field_presence, enum_type, repeated_field_encoding, utf8_validation, message_encoding, json_format`. This is what every helper predicate reads.
3. **FeatureSetDefaults table** — the embedded ordered defaults (LEGACY/PROTO2/PROTO3/2023/2024 entries) as Ball-portable data, plus `minimum_edition`/`maximum_edition`. Authored as a Dart literal so it encodes into the Ball program (no `dart:io`, no asset load at runtime). Refreshed from `protoc --edition_defaults_out` (see §6.1).
4. **Resolution engine** — `resolveFileFeatures(edition, fileFeatures)` and `mergeChildFeatures(resolvedParent, childFeatures)` implementing §3.6 exactly, plus `validateEditionInRange`, `mergeFeatureSet` (MergeFrom semantics), and `isFixedFeature(edition, featureKey)` for override rejection.
5. **Legacy inference** — `inferLegacyFieldFeatures(label, type, packedOption, syntax)` per §3.7, invoked when `syntax` ∈ {proto2, proto3}.
6. **Behavior helpers** — `hasExplicitPresence`, `isRequired`, `isImplicitPresence`, `isOpenEnum`, `isClosedEnum`, `isPackedRepeated`, `isDelimited`, `requiresUtf8Validation`, `jsonFormatIsAllow`. (Existing stub already has some of these; we extend the set and align names with protoc's `has_presence`/`is_packed`/`is_closed`.)
7. **Marshal/unmarshal integration** — extend the field-descriptor map (already used by `marshal.dart`/`unmarshal.dart`/`json_codec.dart`) with an optional `'features'` key (the resolved `Map<String,String>` for that field). Codecs consult it; absence == proto3 defaults (current behavior, zero regression).

### 4.2 How features thread into the existing codecs
The codecs are descriptor-driven. We add `'features'` to each field descriptor and a parallel `'features'` on the message descriptor. Specific touch-points:
- **marshal.dart** (`encodeMessage`): for singular scalar, emit default value iff `hasExplicitPresence(features)` (currently always-omit ≈ IMPLICIT). For repeated scalar, choose packed vs expanded via `isPackedRepeated(features)` (currently hard-coded packed). For message fields, choose LEN vs group via `isDelimited(features)`.
- **unmarshal.dart**: accept both packed and expanded for repeated scalars (already required); for enum fields, if `isClosedEnum(features)` and value out-of-range, route to unknown set instead of storing.
- **json_codec.dart**: gate strict vs best-effort edge handling on `jsonFormatIsAllow(features)`; honor presence for default-value omission.
- **utf8**: string read/write checks `requiresUtf8Validation(features)`.

### 4.3 Data flow

```
                         protoc --edition_defaults_out (build-time, CI)
                                   │
                                   ▼
        embedded FeatureSetDefaults table  ──►  edition.dart / featureset_defaults.dart
                                                          │
 .proto-origin descriptor (FileDescriptorProto)           │
   edition=2023 | syntax=proto3                            │
        │                                                  │
        ▼                                                  ▼
  [legacy inference if syntax]  ──►  resolveFileFeatures(edition, fileFeatures)
                                          │  (5-step algo §3.6)
                                          ▼
                         per-descriptor resolved FeatureSet  (Map<String,String>)
                                          │   attached as 'features' on field/message descriptors
                                          ▼
        ┌─────────── marshal.dart / unmarshal.dart / json_codec.dart ───────────┐
        │  presence • enum open/closed • packed/expanded • delimited • utf8 • json │
        └────────────────────────────────────────────────────────────────────────┘
                                          │
                  (all of the above is Ball-portable Dart source)
                                          ▼
              dart/encoder/bin/gen_ball_protobuf.dart  ──►  dart/shared/ball_protobuf.json / .bin
                                          │
              ┌───────────────────────────┼───────────────────────────┐
              ▼                            ▼                            ▼
        Dart engine                  TS engine                    C++ engine        (same Ball program)
```

The encoder already lists `editions.dart`, so the moment we expand that file the changes flow into `ball_protobuf.json` on the next `gen_ball_protobuf.dart` run.

## 5. Phased Implementation Plan

Each phase is independently reviewable and leaves the tree green.

### Phase 0 — Spec lock & golden-data harness (no engine code)
**Goal:** freeze the spec as machine-checkable golden data so later phases verify against `protoc`, not against our own reading.
- Create: `tests/editions/golden/` — golden resolved FeatureSets for a matrix of `(edition, file-override, message-override, field-override)` cases, generated by a real `protoc` (≥ v32 for edition 2024). Include the legacy-inference cases (proto2/proto3 producing identical resolved sets to equivalent editions files).
- Create: `tools/gen_edition_defaults.ps1` (and `.sh`) — wraps `protoc --edition_defaults_out=...` to emit the `FeatureSetDefaults` binary and a JSON dump, checked in under `tests/editions/featureset_defaults.binpb` + `.json`.
- Create: `docs/EDITIONS_SPEC.md` capturing §3 tables (single source for reviewers).
- **Artifacts:** golden JSON files + a checked-in `protoc --version` note pinning the generator.

### Phase 1 — Edition model + FeatureSetDefaults table (Ball-portable Dart)
**Goal:** represent editions and the embedded defaults table.
- Create: `dart/shared/lib/protobuf/edition.dart` — Edition int constants (§3.1), `editionFromString` / `editionToName`, `syntaxToEdition`.
- Modify: `dart/shared/lib/protobuf/editions.dart` — replace the ad-hoc proto2/edition2023 defaults maps with the full ordered `FeatureSetDefaults` table (LEGACY/PROTO2/PROTO3/2023/2024) derived from Phase 0 golden data; keep the public value constants and existing helper predicates as the stable surface.
- Add to: `dart/shared/lib/protobuf/protobuf.dart` barrel (`export 'edition.dart';`).
- Add to: `dart/encoder/bin/gen_ball_protobuf.dart` `_sourceFiles` → `'lib/protobuf/edition.dart'`.
- **Tests:** `dart/shared/test/protobuf/edition_test.dart` — string⇄int round-trips, defaults-table values match Phase 0 golden.

### Phase 2 — Resolution engine + legacy inference
**Goal:** implement §3.6 and §3.7 in full.
- Modify: `dart/shared/lib/protobuf/editions.dart` — add `mergeFeatureSet`, `resolveFileFeatures`, `mergeChildFeatures`, `validateEditionInRange`, `isFixedFeature`/override-rejection; rewrite `resolveFeatures` to delegate to the file+child path.
- Create: `dart/shared/lib/protobuf/legacy_features.dart` — `inferLegacyFieldFeatures`, `inferLegacyFileFeatures`.
- Add both to the barrel + `gen_ball_protobuf.dart` `_sourceFiles`.
- Expand helper set in `editions.dart` to the full §4.1.6 list.
- **Tests:** `dart/shared/test/protobuf/feature_resolution_test.dart` — drive every Phase-0 golden case through the resolver; assert exact match. Cover: multi-level inheritance, **extension parented by lexical scope not extendee**, **oneof field parenting**, fixed-feature rejection (hard error), and proto2/proto3 inference equivalence.

### Phase 3 — Thread features through binary marshal/unmarshal
**Goal:** make wire encoding honor resolved features.
- Modify: `dart/shared/lib/protobuf/marshal.dart` — read optional `'features'`; presence-gate singular-default emission; packed/expanded selection; delimited (group) message encoding (wire types 3/4).
- Modify: `dart/shared/lib/protobuf/unmarshal.dart` — accept packed AND expanded for repeated scalars; closed-enum → unknown-field routing; group decode for delimited.
- Backward-compat invariant: a descriptor with no `'features'` key behaves exactly as today (proto3 defaults). This guards the existing 100+ conformance/round-trip tests.
- **Tests:** extend `dart/shared/test/protobuf/protobuf_test.dart` with feature-driven round-trips.

### Phase 4 — Thread features through JSON codec + UTF-8
**Goal:** JSON path respects presence, json_format, utf8.
- Modify: `dart/shared/lib/protobuf/json_codec.dart` — presence-aware default omission; `json_format` strict/best-effort branch; UTF-8 validation hook for VERIFY string fields.
- **Tests:** add to `protobuf_test.dart`.

### Phase 5 — Conformance plugin wiring (Dart reference)
**Goal:** the editions logic is reachable from the protobuf conformance harness.
- Modify: `dart/shared/lib/protobuf/conformance.dart` `processConformanceRequest` — route through feature-resolved marshal/unmarshal; at minimum un-skip the proto2/proto3 legacy-inference identity cases.
- **Artifacts:** a `tests/editions/conformance_runner.*` script comparing Ball-engine output to the official protobuf conformance runner.

### Phase 6 — Regenerate the Ball program & prove portability (MILESTONE)
**Goal:** the same Ball editions engine runs on ≥2 targets. *This is the portability proof.*
- Run: `cd dart/encoder && dart run bin/gen_ball_protobuf.dart` → regenerated `ball_protobuf.json` + `.bin` now contain the editions resolver and feature-aware codecs.
- **Target A (Dart):** run resolver/codec via the Dart engine; assert Phase-2 golden FeatureSets resolve identically *through the engine*.
- **Target B (TypeScript):** compile `ball_protobuf.json` through `@ball-lang/compiler`, run same golden cases via `ts/engine`. Assert identical resolved FeatureSets vs Target A.
- **Target C (C++, stretch):** compile through the C++ path; run same golden cases. (C++ engine has a 65 KB linear-memory limit — verify the defaults table + resolver fit; if not, flag as a sizing follow-up rather than blocking the milestone.)
- Create: `tests/editions/portability_matrix.md` — table proving identical resolution output across targets.
- **Tests:** cross-target harness asserting Dart-engine and TS-engine resolution outputs match the golden set.

### Phase 7 — Docs, CI, metadata spec
- Modify: `docs/METADATA_SPEC.md` if any new metadata keys are introduced (likely none).
- Modify: CI (`.github/workflows/ci.yml` + `conformance-matrix.yml`) — add an editions resolution job (Dart + TS, C++ if Phase 6C passed); add the `gen_edition_defaults` regeneration drift check.
- Modify: `CLAUDE.md` / `dart/AGENTS.md` / `.claude/rules/proto.md` — document the editions engine and the "refresh defaults from protoc" workflow.
- **Decision gate:** only here (and only if §7-Q1 is "yes") consider migrating `proto/ball/v1/ball.proto` to `edition = "2023"` — kept strictly separate.

## 6. Testing & Conformance Strategy

1. **Golden FeatureSet resolution (primary correctness gate).** Phase 0 generates resolved FeatureSets with a pinned `protoc`. Every resolver test (Phase 2) and cross-target run (Phase 6) asserts equality against these goldens — guaranteeing we match `feature_resolver.cc`, not our own interpretation.
2. **Embedded-defaults drift check (CI).** `tools/gen_edition_defaults.*` re-derives the table; CI diffs against the checked-in table. Fails loudly if upstream protobuf shifts a default.
3. **Wire round-trips with feature matrices (Phase 3–4).** For each toggled feature, encode→decode→re-encode and assert byte-identity, plus cross-check against `protoc`-produced golden binaries.
4. **Legacy-equivalence tests.** proto2/proto3 descriptors run through inference must produce the *same* resolved FeatureSet and wire output as the equivalent `edition=2023` file (validates §3.7).
5. **Official protobuf conformance harness (Phase 5).** Reuse `conformance.dart`'s size-prefixed protocol against the upstream conformance runner with edition message types; track pass/skip counts as a CI metric.
6. **Cross-target identity (Phase 6, the portability metric).** Same Ball program → identical resolution + codec output on Dart and TS engines; recorded in `tests/editions/portability_matrix.md`.
7. **Regression firewall.** Descriptors without a `'features'` key must keep current behavior — the entire existing `protobuf_test.dart`, `proto_roundtrip_test.dart`, and `tests/conformance/*` suites must stay green after every phase.

## 7. Risks, Unknowns, and OPEN QUESTIONS for the User

1. **Migrate `ball.proto` to editions, or not?** The engine work does NOT require touching `proto/ball/v1/ball.proto` (still `syntax = "proto3"`). Migrating it to `edition = "2023"` is wire-zero-diff *in principle* but triggers `buf breaking` review and regeneration of all 7 bindings. **Recommend: keep ball.proto on proto3 for now; migration is a separate Phase 7 gate.**

2. **Target edition ceiling: 2023, 2024, or 2026?** Edition 2024 adds only source-retention features the *runtime* ignores — so 2024 is nearly free at runtime but needs protoc ≥ v32 for golden data. EDITION_2026 exists but is not a published, fully-specified edition. **Recommend: runtime support for LEGACY/PROTO2/PROTO3/2023/2024; 2026 future/unimplemented.**

3. **How deep does DELIMITED (group) wire support need to go?** Ball programs interchange via proto3-JSON; current codecs are length-prefixed only. Group/delimited (wire types 3/4) is real work. **Question: required v1 deliverable, or documented limitation initially?**

4. **Which 2 targets prove portability first — and is C++ in v1?** Dart + TypeScript are the guaranteed pair. The C++ engine has a 65 KB linear-memory budget which the defaults table + resolver may or may not fit. **Question: Dart+TS sufficient for v1 with C++ fast-follow, or must C++ be in the milestone?**

5. **Embedded defaults source-of-truth & refresh cadence.** We check in a `protoc`-generated `FeatureSetDefaults` and hand-author the equivalent Dart table. **Question: CI *fails* on drift from a pinned protoc (strict, recommended) or just warns? Which protoc version do we pin?**

6. **Custom Ball feature extension — in or out?** We could claim a FeatureSet extension number (≥10000, via a PR to protocolbuffers/protobuf) for a Ball-specific feature. Significant external dependency, almost certainly not needed. **Recommend: out of scope.**
